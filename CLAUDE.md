# Vikram Writing Site Operator Guide

This file is the repo-local operating contract for any coding agent working on `vikramchopra.in`.

## Stack

- Static site generated by `node scripts/build.mjs`
- Source content in `content/`
- Source images in `src/media/`
- Published root output mirrored from `dist/` by `node scripts/sync-root.mjs`
- GitHub Actions handles deploy, site ops monitoring, and monthly reporting

## Core Commands

- `npm run build`
- `npm run publish:root`
- `npm run seo:audit`
- `npm run health:live`
- `npm run perf:budget`
- `npm run qa:visual`

## Builder And Verifier Split

- Builder changes source files only: `content/`, `src/media/`, `scripts/`, workflow files.
- Verifier checks the real artifact:
  - build output in `dist/`
  - root publish output after `npm run publish:root`
  - screenshots for visual work
  - GitHub Actions runs for deploy and site ops when work is pushed
- Do not call a change done until the verifier step has run.

## Review Rules

- Prefer the smallest safe patch.
- Do not edit essay body text unless explicitly asked.
- Treat metadata, redirects, OG images, JSON-LD, and Notes and Sources as production-critical.
- If a change touches output or discoverability, run:
  - `npm run publish:root`
  - `npm run seo:audit`
- If a change touches styles, templates, archive rows, homepage rows, or article layout, also run:
  - `npm run qa:visual`
- For live issues, verify the relevant public URL and the GitHub Actions run after push.

## Code Style

- Preserve the existing plain-JS static-site style.
- Keep logic in `scripts/` explicit and dependency-light.
- Prefer checked-in source assets under `src/media/` over direct root edits in `media/`.
- Avoid adding new npm dependencies unless there is a clear long-term reason.

## Forbidden Or High-Risk Areas

- Do not hand-edit generated `dist/` files as source of truth.
- Do not treat root HTML as canonical when the source lives in `content/` or `scripts/build.mjs`.
- Do not delete or rewrite the large uncommitted creative workspace blindly; it may contain in-progress visual exploration.
- Do not push visual or metadata changes without the verifier step.

## Worktree Rule

- For parallel runs, prefer separate git worktrees.
- Do not run multiple agents in the same checkout when both may edit files.
- Keep one worktree per bounded task: content, visuals, ops, or infra.

## End-Of-Run Rule

- Update `PROJECT_MEMORY.md` with:
  - verified facts
  - failed attempts
  - what changed
  - the next best follow-up

## Incident Closure Rule

For GitHub, deploy, and site-health incidents, "green run" is not enough. Closure requires all of:

- replacement deploy or workflow run is green
- alert issue is closed or recovered
- stale alert email from the incident window is cleared
- lesson is written into `PROJECT_MEMORY.md`

## Inbox Rule

GitHub and site email is part of the operational control plane for this repo.

- Triage fresh GitHub/site mail proactively, not only when asked.
- Treat Gmail as equal in importance to GitHub Actions and GitHub issues.
- Use severity buckets, not one catch-all label:
  - `Ops/GitHub/Security`
  - `Ops/GitHub/CI`
  - `Ops/Site`
  - `Ops/Resolved`
